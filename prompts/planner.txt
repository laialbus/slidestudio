Design a slide plan that covers the key concepts, arguments, and insights from the document. Each slide must have a clear pedagogical purpose.

Return a JSON object with:
- title: concise deck title (max 60 characters)
- total_slides: number of slides planned (between 4 and 20 inclusive)
- slides: list of PlannedSlide objects, each with:
  - index: 1-based slide number
  - tag: one of "Introduction", "Key Concept", "Definition", "Example", "Insight", "Data Point", "Takeaway", "Summary"
  - source_section: the section heading this slide draws from
  - intention: what this slide is meant to teach (max 200 chars)
  - emphasis: what the viewer should walk away remembering (max 200 chars)
  - chunk_indices: list of 1, 2, or 3 integer chunk IDs (NEVER empty, NEVER 4 or more) whose raw text is needed to write this slide accurately

Slide design rules for a college student audience:
- For every 3 "Key Concept" or "Definition" slides, schedule at least 1
  "Example" slide that demonstrates the concept with a concrete case drawn
  from the source material. Do not invent examples.
- When a concept builds directly on an earlier concept, note this dependency
  in the intention field so the Writer knows to briefly anchor the explanation
  to what the student has already seen.
- Schedule "Definition" before "Key Concept" when a term is technical and
  unlikely to be familiar to a college student without prior specialist training.
  Do not assume expert background.
- "Insight" and "Takeaway" slides must express a non-obvious conclusion —
  something a student would not have predicted before reading the material.
  If the slide would merely restate a Key Concept, use that tag instead.
- The final slide must be tagged "Summary" or "Takeaway". Assign it 1–3
  representative chunk IDs like any other slide — do NOT set chunk_indices to [].
  The pipeline detects it by its tag, not by its chunk_indices.

Return ONLY the JSON object. No markdown fences, no explanation.
